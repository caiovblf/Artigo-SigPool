An Educational Laboratory for ECDSA Nonce Reuse Vulnerabilities

The Problem: ECDSA Nonce Reuse

The Elliptic Curve Digital Signature Algorithm (ECDSA), heavily utilized in blockchain networks over the secp256k1 curve, relies on a critical parameter: a cryptographically secure, single-use random value known as the nonce ($k$).

A catastrophic mathematical failure occurs if the exact same nonce $k$ is used to sign two different transactions with the same private key. Because the mathematical structure of ECDSA exposes the internal parameters when $k$ remains static, an external auditor can calculate the nonce and subsequently recover the private key.

The Mathematics of the Exploit

In ECDSA, a signature consists of a pair of integers $(r, s)$. The parameter $r$ is derived directly from the nonce $k$ and the curve's generator point. If the same $k$ is used for two different message hashes ($z_1$ and $z_2$), the resulting signatures will share the exact same $r$ value.

The signature parameter $s$ is calculated as:

$$s = k^{-1}(z + r \cdot d) \pmod n$$

Where $d$ is the strictly confidential private key, and $n$ is the order of the curve. If an attacker possesses two signatures $(r, s_1)$ and $(r, s_2)$ for messages $z_1$ and $z_2$, they can isolate the reused nonce $k$ using simple modular arithmetic:

$$k = (z_1 - z_2)(s_1 - s_2)^{-1} \pmod n$$

Once $k$ is isolated, the attacker trivially extracts the private key $d$, compromising the entire wallet:

$$d = r^{-1}(s_1 \cdot k - z_1) \pmod n$$

Comprehensive Laboratory Scenarios

To study this behavior in an isolated environment, SigPool implements a miniature transactional ecosystem containing a cryptographic engine, a simplified mempool, and a block assembler. The project evaluates five distinct topological scenarios:

1. The Safe Baseline

Two transactions are signed by the same private key using entirely different, cryptographically secure nonces. The resulting signatures are valid, the mempool accepts both transactions, and the external auditor is mathematically unable to correlate the signatures or extract the key.

2. The Nonce Reuse (Critical Vulnerability)

Two distinct transactions are signed by the same key using the exact same nonce. Because the signatures remain mathematically valid regarding the curve, the mempool flawlessly accepts both transactions. However, the background auditor detects the identical $r$ values, executes the extraction algorithm, and successfully recovers the private key.

3. Payload Tampering

A transaction is properly signed, but its payload is maliciously altered post-signature. This scenario proves that the signature no longer matches the message hash ($z$). The mempool correctly rejects the transaction due to an invalid signature, safeguarding the network.

4. Double-Spend Conflict

Two transactions attempt to spend the exact same Unspent Transaction Output (UTXO). The mempool processes them sequentially: the first transaction is accepted, while the second is instantly rejected under the conflict-double-spend rule, preventing ledger corruption.

5. Child Pays for Parent (CPFP)

A demonstration of advanced block assembly mechanics. A "parent" transaction is broadcasted with an exceptionally low fee, followed by a dependent "child" transaction bearing a massive fee. The block assembler evaluates them as a unified package, respecting structural dependencies (parent strictly before child) and optimizing inclusion based on the combined average feerate.

The Paradox of Validity

The most dangerous aspect of this vulnerability is perfectly captured by the block assembler logs. The mempool accepted 2 transactions, and the block assembler successfully packed them into a simplified block weighing 1700 weight units with a combined average feerate of 0.88 sats/vB.

Because the signatures are mathematically flawless, traditional node consensus rules are fully satisfied. The network remains entirely unaware that the owner's private key has been exposed to the public. This simulation fundamentally proves that robust network consensus rules cannot protect users against weak cryptographic implementations at the wallet level.
